# AI 穿搭问答前端重构设计

## 目标

将当前“AI 私人厨师”静态前端重构为以聊天问答为核心的“AI 私人穿搭顾问”，采用编辑式极简与 AI 原生卡片风格，并加入仅用于前端展示的登录弹窗。

本次只重构前端。后端 Python 文件、API 路由、请求字段、OSS 上传流程、流式响应读取方式及历史会话协议全部保持不变。

## 当前项目约束

项目中仅存在 Next.js 静态导出产物，没有原始 React/Next.js 源码与 `package.json`。直接维护压缩后的 JavaScript 不可靠，因此新增一个独立、可维护的前端源码目录，构建完成后将静态导出产物部署到 `app/static`。

## 技术方案

在项目根目录新增 `frontend/`：

- Next.js 静态导出
- React 客户端组件
- TypeScript
- Tailwind CSS 或等效的局部样式体系
- Lucide 图标
- Markdown 渲染保持现有能力

构建输出必须能被 FastAPI 当前的 `StaticFiles` 挂载方式直接提供，不要求修改 `app/main.py`。

## 保持不变的接口契约

前端继续请求 `http://localhost:8001`：

### OSS 签名

```text
GET /api/v1/oss/presign?filename={filename}
```

继续读取：

```json
{
  "uploadUrl": "...",
  "accessUrl": "...",
  "contentType": "..."
}
```

上传仍使用：

```text
PUT uploadUrl
Content-Type: contentType
Body: 原始图片文件
```

### 流式聊天

```text
POST /api/v1/chat/stream
Content-Type: application/json
```

请求体保持：

```json
{
  "message": "...",
  "image_url": "...",
  "thread_id": "..."
}
```

响应继续通过 `response.body.getReader()` 和 `TextDecoder` 逐块读取纯文本，不引入 SSE 或 JSON 分帧。

### 历史与清空

```text
GET /api/v1/chat/messages?thread_id={threadId}
DELETE /api/v1/chat/messages?thread_id={threadId}
```

历史响应继续读取 `messages` 字段。会话 ID 继续存储在 `localStorage` 的 `thread_id`。

## 视觉系统

### 风格

- 编辑式极简
- 问答聊天为主结构
- Bento 卡片用于分析摘要
- 暖白纸张背景，不使用大面积炫彩渐变
- 穿搭照片始终是视觉主角

### 色彩

- 页面背景：`#F7F4EF`
- 主卡片：`#FFFFFF`
- 前景文字：`#211D1A`
- 次级文字：`#746B63`
- 品牌莓红：`#9D174D`
- 浅莓红：`#F8EDEF`
- 边框：`#E4DDD5`
- 深色分析卡：`#211D1A`

### 字体

- 编辑式标题：中文优先使用系统宋体或衬线字体回退
- UI 与正文：系统无衬线字体
- 英文品牌标签使用大写、宽字距

### 形态

- 主容器圆角 20–24px
- 消息气泡圆角 16–18px，并通过单侧小圆角区分对话方向
- 分析卡片圆角 12–16px
- 阴影克制，主要通过边框、留白和背景层次区分
- 所有交互目标至少 44px

## 页面结构

### 顶栏

- 品牌：`ATELIER AI / 私人穿搭顾问`
- 副标题：`上传穿搭照片，获取个性化造型建议`
- 登录按钮
- 新建会话按钮

桌面端保持居中宽容器；移动端压缩副标题，并确保操作按钮仍可触达。

### 空状态

以聊天起始页呈现：

- 主标题：`今天想分析哪套穿搭？`
- 简短说明
- 上传图片主操作
- 三个可点击提示词，如“通勤穿搭”“约会造型”“旅行搭配”

提示词只负责把文本填入输入框或直接发起现有聊天请求，不增加新接口。

### 对话消息

用户消息：

- 右对齐
- 莓红背景与白色文字
- 上传图片显示在文本上方
- 用户圆形头像使用简洁人物图标

AI 消息：

- 左对齐
- 白色纸张卡片
- Markdown 内容保持标题、列表、引用、表格和代码样式
- AI 头像使用品牌字母或衣架图标
- 流式输出时显示细小的“正在分析”状态

### 输入区

- 固定在主内容底部
- 图片选择按钮
- 图片预览与移除按钮
- 文本输入区
- 发送按钮
- 保持 Enter 发送、Shift+Enter 换行
- 上传或生成期间显示禁用和加载状态

### AI 分析卡片

不修改模型输出协议，也不要求后端返回结构化 JSON。常规回答继续以 Markdown 显示。

页面在空状态和流式状态使用静态视觉卡片表达 AI 能力；不对模型文本进行脆弱的自动结构化解析。

## 登录展示

采用方案 A：顶栏登录按钮。

点击后打开前端弹窗，包含：

- 标题：`登录你的穿搭档案`
- 手机号或邮箱输入框
- 密码输入框
- `登录并继续` 按钮
- `微信快捷登录` 展示按钮
- 注册提示

登录不调用任何 API，不发送用户数据。点击登录按钮后仅：

1. 在本地 React 状态中标记为已登录；
2. 顶栏按钮替换为展示头像和昵称；
3. 可选地使用 `localStorage` 保存纯展示状态。

弹窗必须明确这是界面演示或避免制造真实认证成功的误导。刷新后是否保留展示状态由实现选择，以简单可靠为优先。

## 状态与错误处理

- 图片上传失败：在聊天区增加 AI 风格错误消息
- 聊天请求失败：保留已生成内容并追加可读错误提示
- 历史加载失败：不阻塞新会话，控制台记录错误
- 响应流为空：结束加载状态，不展示 `null`
- 登录表单：只进行前端必填状态展示，不进行真实身份校验
- 所有加载、禁用、错误状态需要有可见反馈

## 响应式设计

- 移动端：单列全宽聊天界面，顶栏和输入区使用安全边距
- 平板：主聊天区域最大宽度约 760px
- 桌面：主聊天区域最大宽度约 960px，保留舒适留白
- 图片消息限制高度，避免超出视口
- 长 Markdown 表格允许局部横向滚动，页面本身不产生横向溢出

## 可访问性

- 按钮使用语义化元素和可理解的 `aria-label`
- 弹窗支持 Escape 关闭、遮罩点击关闭和焦点可见
- 文件输入有可访问标签
- 品牌色文字与背景达到至少 WCAG AA 对比度
- 动画遵循 `prefers-reduced-motion`
- 图片提供合理的替代文本

## 文件与部署边界

允许：

- 新增 `frontend/` 源码与配置
- 新增前端测试
- 重建并替换 `app/static` 内的前端静态产物
- 新增前端开发说明文档

禁止：

- 修改 `app/api/**`
- 修改 `app/agents/**`
- 修改 `app/models/**`
- 修改 `app/main.py`
- 修改任何 API URL、字段名或响应解析协议
- 实现真实登录或新增认证后端

## 验证

### 自动验证

- 前端构建成功
- 类型检查通过
- 关键 UI 测试通过
- 静态导出文件存在
- 构建产物中仍包含既有 API 路径和请求字段

### 浏览器验证

- 空状态、图片预览、用户消息、AI 消息和登录弹窗视觉正确
- 登录弹窗只改变本地展示状态
- 新会话仍清空历史并生成新的 `thread_id`
- 响应式检查：移动、平板、桌面
- 键盘操作、焦点状态、禁用状态和错误状态清晰
- FastAPI 提供的页面能正常加载静态资源

## 中断恢复

如果一次执行无法完成，应在 `docs/superpowers/progress/` 保存进度记录，包含：

- 已完成步骤
- 已修改文件
- 已运行验证及结果
- 当前阻塞点
- 下一条应执行的具体命令或修改

该记录用于后续会话从最后一个可靠步骤继续。
